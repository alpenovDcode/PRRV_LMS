lib/tg/* · Архитектура под капотом

Поток входящего сообщения

user → Telegram → POST /api/tg-webhook/[botId]
                  ├─ check X-Telegram-Bot-Api-Secret-Token  (timingSafeEqual)
                  ├─ Redis SETNX(update_id, TTL=1h)         (idempotency)
                  └─ lib/tg/inbound.ts
                       ├─ upsert TgSubscriber
                       ├─ log TgMessage (direction='in')
                       ├─ applyTrackingLink(slug)            (set tags + UTM + firstTouch)
                       ├─ deliverReplyToWaitingRun(text)     (resume flow)
                       ├─ deliverButtonClickToWaitingRun(cb) (goto/tag from button)
                       └─ findTriggeredFlows() → startFlowRun() × N

flow engine state machine:
  queued → running → ┬─ sleeping(resumeAt)     ←── cron tick wakes up
                    ├─ waiting_reply(resumeAt) ←── timeout OR user reply
                    ├─ completed
                    ├─ failed
                    └─ cancelled

Безопасность

  • Токены ботов — AES-256-GCM, ключ из TG_TOKEN_ENC_KEY, в БД лежит зашифрованным
  • Webhook — проверка X-Telegram-Bot-Api-Secret-Token через timingSafeEqual
  • Cron — Bearer-токен из TG_CRON_SECRET, тоже timing-safe
  • Админ-API — withAuth({roles:["admin"]}), JWT в httpOnly cookie
  • HTML — whitelist-санитайзер: <script>, javascript:, onclick уходят как escaped-text
  • Rate-limit — Redis token bucket переживает рестарт и несколько Node-инстансов
  • Idempotency — update_id в Redis SETNX, TG-ретраи не задвоят сообщения

Чего сознательно нет (вне ночи)

  • Visual drag-n-drop редактор графа (есть JSON+Zod-валидация+шаблоны)
  • Multi-tenant — single-tenant, как и остальная ЛМС
  • A/B-тесты, ClickHouse-аналитика, биллинг SaaS
  • WhatsApp / VK / прочие каналы — только Telegram
  • Публичный REST API наружу с API-ключами

Каждое — отдельный слой поверх готового ядра. Data-model и события для них уже подготовлены: события из lib/tg/events.ts можно пайпать наружу для webhooks/BI, шаблоны нод покрывают core-флоу.
