1. Введение и цели

Зачем это построено

Proryv LMS оперирует онлайн-обучением: курсы, рассылки, домашние задания, наставники. До этого MVP воронки и автокоммуникации с лидами/учениками строились через связку из SaleBot (конструктор воронок), BotHelp (рассылки), Битрикс24 (CRM/сделки) и Google Sheets (склейка). Главная боль такой связки — данные передаются между четырьмя системами и постоянно расходятся: статистика не бьётся, кастомные поля нельзя добавить там где нужно, рассылки идут отдельной системой без знания контекста ученика.

Цель MVP — заменить SaleBot и BotHelp на встроенный модуль Bot Platform внутри LMS. Воронки, рассылки, теги, списки, кастомные поля живут в одной БД с курсами, ДЗ и учениками. CRM получает прямой доступ к этим данным без промежуточной синхронизации.

Какие воронки нужно строить

На старте — пять типовых сценариев:

  • Автовебинарные / автоинтенсивные — даты-смещения, напоминания, дожимы
  • Запускные — серия касаний под старт курса
  • Видео-выдача — медиа-сообщения с кружочками и голосовыми
  • Трекинг кликов по ссылкам — ветка «перешёл / не перешёл»
  • Анкеты — сбор email/телефона с regex-валидацией

Что НЕ копировалось из SaleBot

Намеренно опущены фичи, которые в Proryv LMS не нужны или решаются другими модулями:

  • Поддержка прочих мессенджеров (VK/WA/Instagram/Max) — только Telegram
  • Семантическое сравнение фраз через AI — fuzzy / regex закрывают 95% случаев
  • Python-код в Action-ноде — security/sandboxing сложна и не оправдана
  • Сложный CRM-pipelines UI — это отдельный модуль LMS
v2.0 (май 2026) — после первичного MVP добавлены: Iter 6-13 (audit fixes, логи в UI, A/B-эксперименты, funnel-level аналитика, CSV-импорт, schedule-flow, merge subscribers, operator live-chat, 429-retry, health-check, cron heartbeat, ротация секретов, live preview сообщений, семантическая валидация флоу, copy/paste нод, deal-scope viewer, bulk-операции, trigger stats, import/export флоу, hot-reload и +3 шаблона). См. разделы 11–19.

Поток входящего сообщения

Когда подписчик пишет боту, Telegram отправляет update на наш webhook. Дальнейший путь:

  • Verify X-Telegram-Bot-Api-Secret-Token через timingSafeEqual
  • Идемпотентность через Redis SETNX на update_id (TTL 1ч)
  • Upsert TgSubscriber по (botId, chatId)
  • Авто-захват медиа от админов в TgMediaFile (Iter 2)
  • Захват контактов/локации в client.phone, client.location_* (Iter 3)
  • Operator takeover gate (Iter 9) — если оператор взял диалог в ручной режим и сообщение НЕ слэш-команда, ранний return: триггеры и wait_reply не отрабатывают
  • Доставка ответа в активный wait_reply → валидация → onSave bundle
  • Матчинг триггеров активных флоу → start_flow_run с priority/onlyOnce/onlyOnCallback

Поток выполнения флоу

Flow Engine в lib/tg/flow-engine.ts — ядро. Каждый TgFlowRun — конечный автомат со статусами:

СтатусЗначение
queuedГотов исполняться, ждёт ближайший tick
runningВ процессе (короткоживущий)
sleepingВ delay-ноде, проснётся в resumeAt
waiting_replyВ wait_reply, ждёт текст/timeout
completedДошёл до end-ноды
failedОшибка (lastError записан)
cancelledОтменён (вручную, position-change-cancel или hot-reload в Iter 13)

Движение между статусами идёт через два пути:

  • Реактивно — webhook принёс ответ или клик → deliverReplyToWaitingRun / deliverButtonClickToWaitingRun
  • По таймеру — tg-cron sidecar дёргает /api/tg-cron/tick каждые 20с → processDueRuns() подбирает runs с resumeAt ≤ now()

4. Итерация 1 — Фундамент

Самая важная итерация — закладывает позиционную модель (аналог SaleBot-овской «позиции воронки»), безопасный expression-движок и продвинутую систему переменных. Без этого всё остальное невозможно.

Позиционная модель — главное

Каждая message-нода по умолчанию является позицией (isPosition: true). Когда подписчик приходит в позиционную ноду, на нём обновляются currentPositionFlowId/NodeId/At в БД, и отменяются все запланированные (sleeping / waiting_reply) runs, у которых positionGroupId равен предыдущей позиции. Это автоматически реализует «дожимы с автоотменой» — добавил delay-стрелку из Step A в side-effect-сообщение, и оно не отправится, если юзер успел дойти до Step B.

Iter 6 fix: ключ позиции теперь составной — flowId:nodeId (см. positionKey() в lib/tg/flow-engine.ts). Раньше два разных флоу с одинаковым nodeId кросс-отменяли runs друг друга.

Expression Engine

Шаблоны {{...}} в сообщениях и условиях исполняются собственным AST-эвалюатором. Никакого eval(), new Function() или vm.runInNewContext — это убирает целый класс RCE-уязвимостей при копи-пасте чужих флоу.

Эвалюатор состоит из трёх файлов в lib/tg/expr/:

  • tokenizer.ts — лексер на 200 строк, понимает числа, строки, идентификаторы (включая кириллицу), операторы
  • parser.ts — Pratt-парсер с таблицей приоритетов; AST как plain JSON
  • evaluator.ts — обход AST + диспатч в whitelist функций
  • functions.ts — реестр 30+ безопасных функций (даты, строки, regex, массивы)

Regex-валидация в wait_reply

Нода wait_reply поддерживает поле validation с regex-паттерном, кастомным errorMessage (шаблонизируется через expression-движок), и веткой onInvalidNext на случай невалидного ответа. 

validation: {
  pattern: "^[\\w.+-]+@[\\w.-]+\\.[A-Za-z]{2,}$",
  flags: "i",
  errorMessage: "❌ Не похоже на email. Пример: ivan@example.com",
  onInvalidNext: "back-to-ask"
}

Плюс библиотека из 16 готовых regex-пресетов в lib/tg/regex-presets.ts: телефон РФ, email, URL, UUID, дата dd.mm.yyyy, ФИО, и т.д. В UI редактора выбираются одним кликом.

Альбомы

Если в payload-е 2+ медиа-вложения и все они photo или video — sender автоматически отправляет их через sendMediaGroup (альбом). Telegram показывает только подпись первого элемента — это его API, не наше ограничение.

Если медиа смешанные (например, photo + voice) — первый элемент идёт с текстом и кнопками, остальные — отдельными сообщениями fire-and-forget.

Авто-захват file_id

В настройках бота есть поле adminChatIds — массив Telegram chat_id-ов, которым разрешён авто-захват. Когда такой админ отправляет медиа боту:

  • Сообщение не попадает в триггеры и флоу (short-circuit)
  • Медиа извлекается через extractMedia() — для photo берётся самая большая версия
  • Запись в TgMediaFile идемпотентна по (botId, fileUniqueId)
  • Бот отвечает HTML-ack с указанием file_id, размера и kind

Медиа-пикер в редакторе

Компонент components/admin/tg/media-picker.tsx — модальный диалог с превью через media-proxy (/api/admin/tg/bots/[botId]/media/[fileId]). Поддерживает:

  • Поиск по title и filename

  • Pagination через cursor (60 элементов на страницу)
  • Server-side countsByKind для отображения в чипах
  • URL-fallback кнопкой «+ URL» для разовых внешних картинок

6. Итерация 2b — Списки и реактивные триггеры

Списки против тегов

Тэг — свободная строка-метка на подписчике (tags: String[] с GIN-индексом). Идеально для «has_email», «cold», «vip-3-месяца». Создаётся «налету», ничем не предконфигурится.

Список (TgList) — именованный сегмент с id, иконкой, описанием и member-counter. Идеально для «Подписан на интенсив 04-2026», «Активные VIP», «Архив 2024». Это первоклассная сущность для рассылок: в targeting рассылки выбираешь список по id, а не пишешь predicate.

API ops

Все мутации списков идут через lib/tg/lists.ts:

  • addSubscriberToList() — идемпотентно, увеличивает memberCount, фаер'ит list_joined триггеры
  • removeSubscriberFromList() — то же, уменьшает счётчик, фаер'ит list_left
  • moveSubscriberToList() — удалить из A, добавить в B (две транзакции, не атомарно)
  • isSubscriberInList() — для условий

list_joinedПодписчика добавили в список X
list_leftПодписчика удалили из списка X

Реактивные триггеры — мощный инструмент для архитектуры без длинных линейных воронок. Тэг has_email вешается из 5 разных мест (флоу A, флоу B, импорт CSV, ручной ввод оператором), и есть одна воронка-реакция «после сохранения email — приветственное письмо», которая стартует автоматически в любом из этих случаев.

Expression-функции для списков

В условиях и шаблонах доступны:

  • in_list(listId) — true если подписчик в списке (мгновенный lookup, list memberships подгружаются в condition-node)
  • {{lists}} — массив id всех списков, в которых подписчик состоит
  • list_size(listId) — реализовано как заглушка (возвращает 0), для реальных counts — отдельный API

7. Итерация 2c — Типизированные кастомные поля

Зачем

Обычные variables на подписчике — это нетипизированный JSON-bag. Туда можно записать что угодно. Это удобно для разработчиков, но проблемно для оператора, который смотрит на карточку лида и видит хаос разнотипных значений.

Кастомные поля (TgCustomField) добавляют схему: у бота есть набор предопределённых типизированных полей. Каждое поле имеет тип (text/number/date/email/phone/select/boolean/url), опциональную regex-валидацию и флаг isRequired.

8 типов с автовалидацией

ТипCoercion / валидация
textПросто строка до 4096 символов
numberparseFloat с запятой как разделитель
datedd.mm.yyyy / dd-mm-yyyy / dd/mm/yyyy
emailEmail-regex
phoneНормализация: убирает не-цифры, 8 → 7
selectОдин из options[].value
booleantrue/yes/да/1 → true; false/no/нет/0 → false
urlURL-regex (https?://...)

На все типы можно положить ещё один validationRegex — например, email строго в корпоративном домене.

Использование во флоу

Чтобы сохранить ответ в field.x, в wait_reply.saveAs указывается field.<key>. Движок при сохранении смотрит, есть ли в боте поле с таким key, и если есть — применяет валидацию. Если невалидно — отправляется errorMessage, и run идёт по onInvalidNext (или остаётся в waiting_reply ждать новый ответ). 

// На стороне БД пользователя:
TgCustomField {
  key: "phone",
  type: "phone",        // ← автовалидация: 8→7, цифры only
  isRequired: true,
  validationRegex: null
}

// В графе воронки:
{
  type: "wait_reply",
  saveAs: "field.phone",  // ← триггерит валидацию выше
  validation: {           // ← опционально, поверх типа
    pattern: "^\\+?7\\d{10}$",
    errorMessage: "Телефон в формате +79991234567"
  }
}

8. Итерация 3 — UX completion

Reply-keyboard

На каждом сообщении свойство keyboardMode: inline (по умолчанию — кнопки прямо под сообщением с callback'ами), reply (постоянная клавиатура под полем ввода — нажатие отправляет текст кнопки как обычное сообщение), remove (отдать ReplyKeyboardRemove и убрать клавиатуру).

Reply-keyboard — единственный способ запросить телефон (request_contact: true) или геолокацию (request_location: true). В ButtonEditor для этого есть отдельные типы — 📞 Запросить телефон и 📍 Запросить геолокацию.

Авто-захват контакта и локации

Когда подписчик жмёт кнопку «Поделиться телефоном», Telegram присылает message.contact. Inbound handler:

  • Сохраняет client.phone, client.phone_first_name, client.phone_last_name
  • Фаер'ит event subscriber.contact_received
  • Подставляет номер телефона в extractedText, чтобы активный wait_reply считал его обычным ответом и сохранил по своим правилам

С локацией аналогично: client.location_lat и client.location_lon, плюс координаты как extractedText в формате lat,lon.

Redirect-трекинг URL-кнопок

Когда юзер кликает URL-кнопку в сообщении бота, Telegram открывает ссылку напрямую в браузере — мы об этом ничего не знаем. Чтобы получать аналитику кликов, при отправке сообщения движок переписывает все URL-кнопки на короткие /r/<slug>?s=<subscriberId>.

  • На клик: /r/<slug> логирует TgRedirectLink.clickCount, фаер'ит redirect.clicked event, делает 302 на оригинальный URL
  • Slug — 10 base62 символов; per-subscriber переиспользуется
  • Можно отключить per-button через trackClicks: false
  • После этого в условиях работает tag_added('clicked-X') или просто условие по TgEvent в аналитике

Визуальная семантика стрелок

В React Flow редакторе рёбра имеют два стиля:

  • Сплошные серые — переход по действию юзера (нажал кнопку, прислал слово, прошёл регекс)
  • Пунктирные amber-жёлтые animated — переход по времени (delay-нода, timeout wait_reply)

Это визуальная карта воронки: с одного взгляда видно, где юзер «активный», а где система ждёт в фоне. SaleBot-spec называл эту конвенцию обязательной.

9. Итерация 4 — Production hardening

Аудит-лог

Каждое мутирующее действие в админ-API пишется в TgAuditLog с полями: actor (user_id + email), botId, action (формат tg.<entity>.<verb>), details (JSON), outcome (ok/denied/failed), IP, User-Agent. Чтения не пишут — слишком шумно.

Обёртка withAdminApi() в lib/tg/admin-api.ts подключает аудит автоматически при 2xx-ответе: достаточно прописать auditAction при определении endpoint-а. 

return withAdminApi(request, async (req) => {
  // ... business logic
}, {
  rateLimitScope: "write",
  auditAction: "tg.flow.update",
  botId: params.botId,
});

Rate-limit

Per-user token-bucket в Redis. Три scope-а с разными лимитами:

ScopeRefillBurstПрименение
default10 req/sec30 запросовGET-эндпоинты
write4 req/sec12 запросовPOST/PATCH/DELETE
broadcast0.2 req/sec2 запросаСоздание рассылок

При превышении возвращается 429 Rate Limited с заголовком Retry-After. При недоступности Redis rate-limit fails OPEN — лучше пропустить пару лишних запросов, чем заблокировать админку из-за инцидента с кэшем.

Безопасность

Меры защиты:

  • Токены ботов шифруются AES-256-GCM при сохранении в БД
  • Webhook проверяет X-Telegram-Bot-Api-Secret-Token через timingSafeEqual
  • Идемпотентность update-ов через Redis SETNX (TTL 1 ч)
  • HTML-санитайзер для исходящих сообщений: whitelist <b>, <i>, <u>, <s>, <a>, <code>, <pre>, <span>
  • Авто-захват медиа работает только из chat_id-ов в whitelist bot.adminChatIds
  • Expression-движок — без eval, без host-функций
  • 152-ФЗ: вся PII (телефоны, email, имена) хранится в локальной БД клиента

10. Итерация 5 — Компактные воронки (inline-действия)

Проблема: воронка из 5 шагов = 25 нод

Классическая SaleBot-подобная модель: каждое атомарное действие (add_tag, set_variable, add_to_list) — отдельная нода на холсте. В реальной воронке из 5 шагов между ними висит 15-20 узких side-effect-нод. Холст превращается в кашу.

Решение: inline-действия

Узкие действия теперь живут как inline-настройки на родительском узле через бандл InlineActions:

Где живётКогда исполняется
message.payload.onSendПосле успешной отправки сообщения
wait_reply.onSaveПосле сохранения ответа + валидации
button.onClickПри клике на callback-кнопку
actions-нода (отдельная)Fallback на редкий случай side-effect без сообщения

Бандл содержит 5 типов атомарных операций:

  • addTags: string[] — добавить тэги
  • removeTags: string[]
  • addToLists: string[] — id списков
  • removeFromLists: string[]
  • setVariables: {key, value, asExpression?}[]

Порядок исполнения детерминированный:

  • setVariables — первыми, чтобы последующие могли использовать через шаблоны
  • addTags → removeTags → addToLists → removeFromLists

Палитра редактора

С Iter 5 палитра сокращена с 13 до 9 типов нод. Старые узкие ноды (add_tag, remove_tag, add_to_list, remove_from_list, set_variable) остались в схеме для обратной совместимости с уже сохранёнными флоу, но удалены из палитры. Новые флоу собираются через inline.

Чип ⚙ N на нодах canvas

На каждой message- и wait_reply-ноде в редакторе показывается чип «⚙ N» — количество inline-действий. Видно с одного взгляда, что эта нода не только отправляет сообщение, но ещё что-то делает.

onClick — техническая сторона

Telegram callback несёт только callback_data (до 64 байт). Если у кнопки задан onClick, движок генерит синтетический callback вида act:<rowIdx>:<colIdx>. При получении callback'а из webhook'а handler ищет исходное сообщение через tg_message.tgMessageId, оттуда берёт sourceId = flowId:nodeId, парсит граф, достаёт node.payload.buttonRows[row][col].onClick и исполняет.

Iter 6 fix: button-onClick теперь может писать в deal.x-переменные. Раньше путь callback не передавал runId в executeInlineActions, и записи в deal-scope молча падали (см. NPS-тест 5⭐ → detractor).

11. Итерация 6 — Audit fixes + страница «Логи» NEW

После прохождения первого продакшен-цикла обнаружились три критичных бага в движке и был добавлен отдельный UI для просмотра событий.

Bug 1 — Cross-flow position collision

positionGroupId хранил только nodeId. Два разных флоу с нодами одинакового id («msg-1») кросс-отменяли runs друг у друга. Фикс: составной ключ positionKey(): 

export function positionKey(
  flowId: string | null | undefined,
  nodeId: string | null | undefined
): string | null {
  if (!flowId || !nodeId) return null;
  return `${flowId}:${nodeId}`;
}

Bug 2 — /start eaten by wait_reply

Юзер в email-wait_reply пишет /start → раньше попадал в валидатор «❌ Не email» и был заблокирован. Фикс: в handleUserMessage() слэш-команды (распознаются через parseCommand()) пропускают deliverReplyToWaitingRun и идут сразу в trigger-matcher.

Bug 3 — Deleted-flow cron churn

Удаление флоу не отменяло его sleeping runs. Tick по resumeAt подтягивал их → находил !flow → пометит run failed, но через короткое время другой tick снова подбирал runs со stale resumeAt. Фикс: processDueRuns теперь явно фильтрует runs у живых флоу + при попытке выполнить !flow сразу cancel вместо fail.

Страница «Логи» в админке

Новый эндпоинт /api/admin/tg/bots/[botId]/logs + страница /admin/bots/[botId]/logs — просмотр TgEvent с фильтрами:

  • Поиск по типу события и payload-у (JSON contains)
  • Фильтр по уровню (info / warn / error — выводится из типа события)
  • Фильтр по subscriber-id, flow-id
  • Cursor-pagination
  • Русскоязычный каталог событий (32 типа): «Отправлено сообщение», «Получен ответ», «wait_reply таймаут», «Отказ от ретрая», и т.д.

Сам flow-engine при trackEvent теперь добавляет в properties контекст для UI: для message.send_failed пишет classification (retry/blocked/fatal) и source (flow/manual/broadcast), для flow.node_not_found — какая нода и предыдущая, для token_decrypt.failed — отдельный hint с «проверь TG_TOKEN_ENC_KEY».

12. Итерация 7 — A/B split + funnel-level аналитика NEW

Split-нода

Новый тип ноды для A/B (или N-way) тестирования. Каждой ветке задан weight (целое ≥ 1). Engine складывает все веса, бросает random в этом диапазоне, выбирает ветку первого попадания (weighted random).

ВесаРаспределение
[1, 1]50/50
[3, 1]75/25
[1, 1, 1, 1]по 25% каждая

Выбранный label ветки записывается в run.context._abVariant. Downstream condition-ноды могут ветвиться по варианту (deal._abVariant == 'A'), а аналитика агрегирует по нему.

В редакторе split-нода имеет per-branch handle справа — каждая ветка тянется в свой next отдельной стрелкой.

Подсветка unreachable нод

Редактор делает BFS от триггерной псевдо-ноды по графу. Ноды, не достижимые из триггера, рендерятся полупрозрачными (opacity: 0.4) + grayscale. Сверху холста баннер: «Недостижимые ноды: 3 — не подключены к графу через стрелки».

Funnel-summary endpoint

GET /admin/tg/bots/[botId]/analytics/funnel-summary?period=... — сводная таблица по всем флоу с метриками:

  • entered — уникальные подписчики, у которых flow.node_executed хитнул startNode флоу
  • completed — уникальные с событием flow.completed
  • conversionRate = completed / entered × 100
  • medianTimeSec — медиана длины run-а от первого node_executed до completed (P50 через PERCENTILE_DISC(0.5))

A/B-variants endpoint

GET /admin/tg/bots/[botId]/analytics/ab-variants?flowId=... — для каждого split-узла во флоу:

  • Список вариантов с entered (из flow.ab_split) и completed (join по subscriber_id с flow.completed)
  • conversionRate per variant
  • winner — label с максимальной конверсией (если ≥ 1 завершение в выборке)

UI: расширенная вкладка «Воронки» аналитики — сверху сводная таблица по всем флоу с цветовой подсветкой низкой конверсии, снизу выбранного флоу — секция «A/B-эксперименты» с горизонтальным bar-chart’ом по каждому split-узлу, победитель помечен 🏆 trophy-иконкой.

Статистическая значимость. Для уверенности в победе варианта нужно ≥ 50–100 подписчиков на ветку. Меньше — шум. UI показывает абсолютные числа рядом с процентами, чтобы это было видно.

15. Итерация 10 — Операционные риски NEW

Серия мер «закрыть сюрпризы в продакшене» — то, что обычно ломается ночью с пятницы на субботу.

429 retry + retry-after

callApi в lib/tg/api.ts переписан с поддержкой retry:

  • На 429: один inline-ретрай с уважением retry_after, но только если ожидание ≤ 10 сек (иначе возвращаем ошибку наверх — пусть broadcast worker планирует на nextAttemptAt с правильной отсрочкой). Случайный джиттер 0–200 мс сверху, чтобы параллельные процессы не били ровно в одну ms.
  • На 5xx и сетевые ошибки: до 2 ретраев с linear backoff (300 мс × попытка).

В SendResult теперь пробрасывается retryAfterSec. Broadcast worker берёт max(свой backoff, retry_after + 5с буфер) вместо фиксированных задержек — не долбит API раньше времени и не провоцирует новые 429.

Bot-токен. POST /bots/[botId]/rotate-token:

  1. Валидирует новый токен через getMe
  2. Требует тот же bot_user_id — защита от случайной перецепки на чужого бота
  3. Переустанавливает webhook на новом токене (с тем же webhookSecret) до записи в БД

19. Async AI-checker для homework NEW

Не часть бот-платформы напрямую, но связано — это альтернативный pattern для длительных внешних запросов, которые применим и для других long-running операций (AI-генерация ответов и т.п.).

Зачем async

Claude Code CLI на одно задание (особенно с картинками) тратит 2–5 минут. ngrok-free режет любое HTTP-соединение на 2 минутах. Синхронный POST к AI-checker → client_timeout / upstream_timeout срабатывает раньше, чем CLI заканчивает.

Решение — kickoff + callback. LMS даёт AI-checker задание и URL для callback, AI-checker сразу подтверждает приём (HTTP 202), считает в фоне, потом POST-ит результат на callback.

ai_analysis_errorТекст ошибки, если AI-checker не смог обработать (или связь оборвалась)

Полная спецификация в docs/AI_CHECKER_CONTRACT.md. Ключевые требования:

  • Принимать POST с payload и callbackUrl + callbackSecret
  • Возвращать 202 в течение 15 секунд
  • POST-ить результат на callbackUrl с тем же secret в заголовке

23. Диагностика проблем

«Бот не отвечает»

  • Открой /admin/bots/<id> → карточка «Состояние бота в Telegram» (Iter 10). Должно быть зелёным
  • Если TOKEN_MISMATCH — кто-то засунул в БД токен от другого бота. Используй ротацию (карточка «Ротация секретов»)
  • Если WEBHOOK_NOT_SET или WEBHOOK_DRIFT — кнопка «Переустановить webhook»
  • Если ошибка Unauthorized 401 от Telegram — токен ротировали. Через карточку «Ротация секретов» вставь новый токен

«Таймауты wait_reply не срабатывают» / «дожимы не уходят»

Самая частая причина — не настроен tg-cron sidecar.

  • Открой /admin/bots/<id> → карточка «Состояние крона» (Iter 10). Если красная — крон не пингует
  • Проверь логи: docker compose logs --tail=10 tg-cron — должны быть строки с {"ok":true,...}
  • Если sidecar не поднялся: docker compose ps tg-cron — статус должен быть Up
  • Если TG_CRON_SECRET пустая → endpoint возвращает 503. Сгенерируй: openssl rand -hex 32
  • Если sidecar бьёт но получает 401 — TG_CRON_SECRET в шелле и в .env не совпадают
  • Ручной тест: curl -X POST -H "Authorization: Bearer $TG_CRON_SECRET" https://prrv.tech/api/tg-cron/tick

«Telegram возвращает 429 / Too Many Requests» NEW

  • Это не ошибка, это normal rate-limiting. С Iter 10 движок автоматически ретрает с retry_after при ожидании ≤ 10с
  • Если retry_after > 10с (TG наказал нас за burst) — broadcast worker сам отложит на now + retry_after + 5с буфер
  • Проверь страницу «Логи» — события message.send_failed с classification=retry
  • Если 429 постоянны — посмотри частоту рассылок, может стоит дробить большие на батчи

«Рассылка зависла в sending»

  • Проверь cron — он мог упасть посредине (карточка «Состояние крона»)
  • В БД: UPDATE tg_broadcasts SET status='draft' WHERE id='...'; и пересоздай

«Токены ботов перестали расшифровываться»

Симптом: invalid auth tag в логах. Корень: ротировали TG_TOKEN_ENC_KEY без миграции.

  • Откатить env обратно — если старый ключ ещё помнишь
  • Если потерян безвозвратно — токены ботов вводить заново через интерфейс создания (или ротации, если бот уже создан)

«В медиа-библиотеке пусто, хотя я кидал админу медиа»

  • Проверь, что твой chat_id в bot.adminChatIds в БД
  • Узнать свой chat_id: написать @userinfobot
  • Добавить через: /admin/bots/<id> → карточка «Авто-захват медиа»

«Кнопки не реагируют»

  • URL-кнопки работают всегда. Проблема обычно с callback'ами
  • Проверь что webhook на боте установлен (карточка «Состояние вебхука»)
  • Проверь логи приложения: ищи callback_query или btn:noop

«Saved flow и куча failed runs после редактирования» NEW

С Iter 13 hot-reload runs мониторит исчезновение нод и cancel-ит затронутые runs ещё на этапе PATCH:

  • В UI редактора перед сохранением показывается confirm с количеством затронутых runs
  • Если случайно сохранил без confirm — посмотри в «Логи» событие flow.hot_reload_cancelled_runs для аудита
  • Runs со статусом cancelled и lastError = "node removed during hot-reload" — это они, а не баг

«В сайдбаре чата активные сценарии без deal-vars» NEW

  • Iter 12 показывает run.context в раскрывающейся строке. Если поле пусто — у run-а ещё нет deal-переменных, это норма для свежих runs
  • Если ждёшь A/B-вариант: _abVariant попадает в контекст только после прохождения split-ноды

«Я случайно нажал bulk-action — как откатить?» NEW

  • Откат точечно: для add_tag → запусти remove_tag с тем же набором
  • start_flow откатить нельзя — runs уже создались. Если они в sleeping/waiting_reply, можно массово cancel через SQL
  • block легко обратим через unblock-вариант той же bulk-операции
  • Все bulk-действия пишут в TgAuditLog как subscribers.bulk_action — там видна выборка
